## `src/config/` — Source of Truth

```
src/config/
├── app.config.ts              ← modules, auth, locale, payment
├── branding.config.ts         ← name, logo, colors (reads env)
├── permissions.config.ts      ← permission matrix
└── index.ts                   ← barrel export
```

Principle: change the config = change the behavior. Don't hard-code in components.

## `src/components/ui/` — shadcn/ui

Components are copied into the repo (not npm dependencies). Full control of styling & behavior.

```bash
npx shadcn@latest add dialog
```

The component lands in `src/components/ui/dialog.tsx`. Edit freely — it's your code, not vendor code.

## `src/proxy.ts` — Middleware

Named "proxy" (not "middleware") because Next.js reserves `middleware.ts` at the root. In this boilerplate, the root `middleware.ts` re-exports `proxy` so the middleware function still runs.

`proxy.ts` responsibilities:

- Refreshes the Supabase session on every request (SSR)
- Redirects unauthenticated users to `/login`
- Sets the `returnTo` query param for deep-linking after login
- Allows public routes (`/login`, `/register`, auth API endpoints)

## Import Conventions

**Path alias** `@/*` → `./src/*`. Avoid deep relative paths (`../../../`).

```ts
// ✅ Preferred — absolute import via alias
import { useAuthStore } from "@/core/auth/store";
import { FullPageLoader } from "@/core/components";

// ❌ Avoid — deep relative
import { useAuthStore } from "../../../core/auth/store";
```

## Naming Conventions

- **File**: `kebab-case.ts` (e.g. `use-auth.ts`, `profile.service.ts`)
- **Component**: `PascalCase` exported from a kebab-case file
- **Hook**: `use-` prefix
- **Service**: `.service.ts` suffix
- **Store**: `-store.ts` suffix
- **Types**: `types.ts` or `<domain>.types.ts`

## Dependency Rule

**Inward only.** Higher layers may import from lower layers:

- `app` → imports from `modules`, `core`, `config`
- `modules` → imports from `core`, `config`, `shared`
- `core` → imports from `config` only
- `config` → zero dependencies

Modules **may not directly depend on each other**. Commerce can't import from admin. If shared logic is needed, move it to `core` or `shared`.

Enforcement: convention + code review. No lint rule enforces this — you can add one via `eslint-plugin-import` + a custom `no-restricted-imports` rule.

## Barrel Export Pattern

Most folders have an `index.ts` barrel:

```ts
// src/core/auth/index.ts
export { LoginForm, RegisterForm, LogoutButton } from "./components";
export { useAuth, usePermission } from "./hooks";
export { can, canAll, canAny, canAccessAdmin } from "./lib";
export { fetchActiveProfile, verifyProfile } from "./services";
export { useAuthStore } from "./store";
```

Consumer imports:

```ts
import { useAuth, LoginForm } from "@/core/auth";
// or deep import for optimal tree-shaking
import { useAuth } from "@/core/auth/hooks";
```

Deep imports are still encouraged for performance — barrel exports sometimes inflate bundle size if the bundler's tree-shaking isn't optimal.
