Total: **18 env vars.**

| Category | Count | Required |
|---|---|---|
| Supabase | 3 | All required |
| Encryption | 1 | Required |

| App URL | 1 | Optional locally, required in production |

**Minimum viable `.env.local`:**

```bash
# Supabase
NEXT_PUBLIC_SUPABASE_URL=https://xxxxxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGc...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGc...

# Encryption
ENCRYPTION_KEY=<base64 32 bytes>

# Resend (auth email)
RESEND_API_KEY=re_xxxxxxxxxxxxx
RESEND_FROM_EMAIL="My App <noreply@yourdomain.com>"

# Supabase Auth Hook (Send Email Hook)
SUPABASE_AUTH_HOOK_SECRET=v1,whsec_xxxxxxxxxxxxx
```

Seven essential lines. Branding is optional (has defaults). `NEXT_PUBLIC_APP_URL` is optional locally.

## 1. Supabase (3 variables)

Used in: `core/lib/supabase/{client,server,proxy,service-role}.ts`, `app/layout.tsx` (preconnect hint).

### 1.1 `NEXT_PUBLIC_SUPABASE_URL`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | Yes (`NEXT_PUBLIC_` prefix) |
| Default | — (app crashes if empty) |
| Format | Full URL with `https://` |
| Example | `https://abcdefgh12345.supabase.co` |

**Where to find it:** Supabase Dashboard → project → **Project Settings** (gear icon) → **Data API** → **Project URL**.

**Used for:**
- Browser Supabase client via `createBrowserClient`
- Server Supabase client (RSC, Route Handlers) via `createServerClient`
- Middleware proxy (session refresh)
- Service-role client (webhook ingestion)
- `<link rel="preconnect">` in the root layout to speed up TLS handshake

### 1.2 `NEXT_PUBLIC_SUPABASE_ANON_KEY`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | Yes |
| Default | — |
| Format | JWT (starts with `eyJ...`) |
| Example | `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.xxxxx` |

**Where to find it:** Dashboard → **Project Settings** → **Data API** → **Project API keys** → `anon` `public` row → copy.

**Safe to expose?** Yes. The anon key is designed for client-side use. RLS (Row Level Security) enforces permissions — the anon key cannot bypass RLS. If you have tables or storage that should be public, configure it through RLS policies.

### 1.3 `SUPABASE_SERVICE_ROLE_KEY`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | **NO** (no `NEXT_PUBLIC_` prefix) |
| Default | — |
| Format | JWT |
| Example | `eyJhbGc...` (different value from the anon key) |

**🔒 CRITICAL — DO NOT LEAK:**

- The service role key **bypasses RLS**. Anyone with this key can read/write any table without auth checks.
- **DO NOT** commit it to Git
- **DO NOT** prefix it with `NEXT_PUBLIC_`
- If leaked: Dashboard → Project Settings → API → **Reset** service_role → update the env in every environment

**Where to find it:** Dashboard → **Project Settings** → **Data API** → **Project API keys** → `service_role` `secret` row → **Reveal** → copy.

**Used in:** `core/lib/supabase/service-role.ts` — consumed by the `POST /api/commerce/webhooks/[token]` route handler. Lemon Squeezy webhooks arrive **without a user session**, so anon+RLS can't be used. The service role upserts into `commerce_webhook_events`, `commerce_orders`, and `commerce_subscriptions`.

## 2. Encryption (1 variable)

### 2.1 `ENCRYPTION_KEY`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | **NO** |
| Default | — (throws if empty) |
| Format | Base64-encoded 32 bytes |
| Example | `GJ4p8QzN5xK2mH9vL3sW7yT1bE6dA0cF+RiU8oZpVnM=` |

**What it's for:**

`core/lib/encryption.ts` — AES-256-GCM encryption. Used to encrypt:

- LS API keys before storing them in `commerce_credentials.encrypted_api_key`
- Webhook secrets before storing them in `commerce_webhook_configs.encrypted_secret`

**How to generate:**

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

Copy the output into `.env.local`:

```bash
ENCRYPTION_KEY=GJ4p8QzN5xK2mH9vL3sW7yT1bE6dA0cF+RiU8oZpVnM=
```

### ⚠️ Critical Rules

1. **The value must be exactly 32 bytes** after base64 decoding. If it's a different size, `getKey()` in `encryption.ts` throws:
   > `ENCRYPTION_KEY must be 32 bytes (base64 encoded). Got N bytes.`

2. **The value MUST be the same across all environments** that share the same database (dev / staging / prod). Different values mean encrypted data in one env can't be decrypted in another.

3. **Back it up to a secret manager** (1Password, Bitwarden, AWS Secrets Manager). **Losing the key means losing all encrypted data.** There's no recovery.

4. **Rotation:** If you need to rotate the key (e.g. after a leak):
   - Decrypt every row with the old key
   - Re-encrypt with the new key
   - Update the env with the new key
   - There's no auto-rotation mechanism in the boilerplate

## 3. Email — Resend (2 variables)

Used for all auth emails (magic link, recovery, signup confirmation, email change, reauthentication) via the Supabase Send Email Hook.

### 3.1 `RESEND_API_KEY`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | **NO** |
| Default | — (crashes when the hook fires) |
| Format | `re_xxxxxxxxxxxxx` |
| Example | `re_AbCdEfGhIjKlMnOp1234567890` |

**Where to get it:**

1. Sign up at [resend.com](https://resend.com)
2. Dashboard → **API Keys** → **Create API Key**
3. Name is freeform (e.g. `my-app-production`)
4. Permission: **Full access** or **Sending access**
5. Copy the key (⚠️ it appears only once — if you lose it, create a new one)

**Used in:** `src/shared/email/resend-client.ts` — `getResendClient()` lazy-initializes the Resend SDK as a singleton.

**Rotation:** Resend Dashboard → create a new key → update the env → redeploy → delete the old key.

### 3.2 `RESEND_FROM_EMAIL`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | **NO** |
| Default | — |
| Format | `"Sender Name <email@domain.com>"` (RFC 5322) |
| Example | `"Acme Platform <noreply@acme.com>"` |

**Important — the domain must be verified in Resend:**

- The sender email must use a domain that's **verified** (DKIM + SPF configured)
- Unverified domain → Resend returns 400 "Domain is not verified"
- Sandbox alternative for dev: `onboarding@resend.dev` (can only send to the email used to register the Resend account)

**Valid formats:**

```bash
# ✅ Correct — with sender name
RESEND_FROM_EMAIL="My App <noreply@yourdomain.com>"
RESEND_FROM_EMAIL="Acme <hello@acme.com>"

# ✅ Correct — email only (not recommended, empty sender name)
RESEND_FROM_EMAIL=noreply@yourdomain.com

# ❌ Wrong — domain not verified
RESEND_FROM_EMAIL="Me <email@unverified-domain.com>"

# ❌ Wrong — free provider (Gmail, Yahoo won't work)
RESEND_FROM_EMAIL=myapp@gmail.com
```

**Used in:** `src/shared/email/resend-client.ts` — `getFromAddress()`.

**Best practices:**

- Match the sender name to your brand name (`brandingConfig.name`)
- Use a prefix like `noreply@` / `hello@` / `support@`
- Don't use a personal email (`john@company.com`) — looks unprofessional for transactional mail

## 4. Supabase Auth Hook (1 variable)

### 4.1 `SUPABASE_AUTH_HOOK_SECRET`

| Property | Value |
|---|---|
| Required | **Yes** |
| Exposed to browser | **NO** |
| Default | — (hook returns 500 "server_misconfigured" if empty) |
| Format | `v1,whsec_xxxxxxxxxxxxx` |
| Example | `v1,whsec_AbCdEfGhIjKlMnOpQrStUvWxYz1234567890` |

**Naming — canonical vs legacy:**

The boilerplate's webhook verifier at `src/shared/email/verify-webhook.ts` reads from this variable first, with a fallback to the legacy name `SEND_EMAIL_HOOK_SECRET`:

```ts
const secret =
  process.env.SUPABASE_AUTH_HOOK_SECRET ??  // canonical — prefer this
  process.env.SEND_EMAIL_HOOK_SECRET;       // legacy alias — still supported
```

**Use `SUPABASE_AUTH_HOOK_SECRET` for all new deployments.** It matches the nomenclature used in the Supabase dashboard and other Supabase docs. The legacy fallback exists so older deployments don't break during upgrade — you can remove the fallback in `verify-webhook.ts` once every environment has migrated to the canonical name.

**Where to get it:**

1. Supabase Dashboard → **Authentication** → **Hooks** → **Send Email Hook**
2. Enable the hook → Type **HTTPS** → fill in the hook URL
3. Click **Generate secret** → Supabase auto-generates it and displays it below the Save button
4. Copy the **EXACT** value (including the `v1,` prefix)

Full setup details in [Supabase Setup step 6](/docs/setup/supabase#6-send-email-hook-critical-for-auth-email).

**Format breakdown:**

- `v1,` — Standard Webhooks protocol version (**required prefix**)
- `whsec_` — standard webhook-secret prefix
- Remaining characters — random secret

**If you copy the value without the `v1,` prefix:** HMAC verification fails → the hook returns 401 "invalid_signature".

**Used in:** `src/shared/email/verify-webhook.ts` — `verifyHookRequest()` via the `standardwebhooks` library.

**Rotating the secret:**

1. Supabase Dashboard → Auth → Hooks → **Regenerate secret**
2. Update the env → redeploy
3. **No downtime** because the old secret is invalidated immediately when regenerated

Official reference: [Supabase docs — Send Email Hook](https://supabase.com/docs/guides/auth/auth-hooks/send-email-hook).

## 5. App URL (1 variable)

### 5.1 `NEXT_PUBLIC_APP_URL`

| Property | Value |
|---|---|
| Required | Production **required**, local **optional** |
| Exposed to browser | Yes |
| Default | — (falls back to `new URL(request.url).origin`) |
| Format | Full URL with scheme, **no** trailing slash |
| Example | `https://my-app.vercel.app` or `http://localhost:3000` |

**Used in two places:**

1. **`src/shared/email/resend-client.ts` → `getAppUrl()`** — to build the verification URL in emails:
   ```
   {APP_URL}/api/auth/confirm?token_hash=xxx&type=yyy&next=zzz
   ```

2. **`src/app/api/commerce/webhooks/config/route.ts` → `getAppUrl()`** — to construct the per-user webhook URL:
   ```
   {APP_URL}/api/commerce/webhooks/{token}
   ```

**When it's required:**

- **Production:** required so URLs use your production domain (instead of falling back to request-header origin, which can be wrong behind a proxy chain)
- **Local dev:** can be omitted — the fallback to `http://localhost:3000` from the request origin works correctly

**Format:**

```bash
# ✅ Correct
NEXT_PUBLIC_APP_URL=https://my-app.vercel.app
NEXT_PUBLIC_APP_URL=http://localhost:3000

# ❌ Wrong — trailing slash
NEXT_PUBLIC_APP_URL=https://my-app.vercel.app/

# ❌ Wrong — no scheme
NEXT_PUBLIC_APP_URL=my-app.vercel.app
```

The code auto-strips trailing slashes (`replace(/\/$/, "")`), but stay consistent anyway.

## 6. Branding (10 variables)

All optional. All have defaults in `src/config/branding.config.ts`. Used to customize branding without touching code.

### 6.1 `NEXT_PUBLIC_APP_NAME`

| Property | Value |
|---|---|
| Required | No |
| Default | `"My App"` |
| Used in | Metadata title, meta description, PWA manifest `name`, welcome message, login header, **email template branding** |

```bash
NEXT_PUBLIC_APP_NAME="Acme Platform"
```

### 6.2 `NEXT_PUBLIC_APP_SHORT_NAME`

| Property | Value |
|---|---|
| Required | No |
| Default | `"App"` |
| Used in | Sidebar logo text, compact header, PWA `short_name` (homescreen icon), apple-mobile-web-app-title, metadata title template (`%s \| ${shortName}`), **email subject line** |

```bash
NEXT_PUBLIC_APP_SHORT_NAME="Acme"
```

Email subjects are hardcoded Indonesian and use `shortName` for substitution:

- `"Masuk ke {shortName}"` (magic link)
- `"Reset password {shortName}"` (recovery)
- `"Konfirmasi perubahan email {shortName}"` (email change)
- `"Kode verifikasi {shortName}"` (reauthentication)

> To ship English subjects, edit the `getSubject()` switch statement in `src/shared/email/send-auth-email.tsx`.

**Why separate from `APP_NAME`?** PWA `short_name` must be short (max ~12 chars so it doesn't truncate on the homescreen), and email subject lines should also stay concise.

### 6.7 `NEXT_PUBLIC_APP_PRIMARY_COLOR`

| Property | Value |
|---|---|
| Required | No |
| Default | `"#16a34a"` (Tailwind green-600) |
| Format | Hex color with `#` prefix |
| Used in | `<meta name="theme-color">`, PWA `theme_color`, `msapplication-TileColor`, splash screen background, **email template accent (buttons, links)** |

```bash
NEXT_PUBLIC_APP_PRIMARY_COLOR="#2563eb"
```

### ⚠️ Used in Two Layers — Manual Sync Required

1. **Browser chrome / PWA / email:** this env value is used directly (green default).
2. **UI primary (Tailwind):** defined in `src/app/globals.css` as a CSS variable `--primary` in **OKLCH** format. The shipped value is a **dark navy** (`oklch(0.208 0.042 265.755)`) — **NOT** green — and **must be synced manually** when the primary color changes. There's no auto-sync.

Result: out of the box, PWA theme-color and email buttons are green, but the in-app UI primary is navy. Update both layers to keep them aligned.

To change the UI primary fully:

1. Update `NEXT_PUBLIC_APP_PRIMARY_COLOR` (for browser chrome + email)
2. Edit `--primary` in `src/app/globals.css` — convert hex to OKLCH via [oklch.com](https://oklch.com)

### 6.8 `NEXT_PUBLIC_APP_BG_COLOR`

| Property | Value |

| Used in | PWA manifest `background_color` (splash screen background when the PWA launches from the homescreen) |


**When to diverge from white?** If your app defaults to dark mode — a white splash screen is jarring against a dark UI. Match your dominant theme.

### 6.10 `NEXT_PUBLIC_APP_LANG`

| Property | Value |
|---|---|
| Required | No |
| Default | `"id-ID"` |
| Format | BCP-47 language tag |
| Used in | PWA manifest `lang`, HTML `<html lang="...">` |

**Valid examples:**

- `id-ID` — Indonesian (Indonesia) — **default**
- `en-US` — English (US)
- `en-GB` — English (UK)
- `ja-JP` — Japanese
- `zh-CN` — Simplified Chinese

```bash
NEXT_PUBLIC_APP_LANG="en-US"
```

**Note:** this is **not** the UI locale. The UI locale is controlled via `appConfig.locale.default` in `src/config/app.config.ts` (value `"id"` or `"en"`). `NEXT_PUBLIC_APP_LANG` is only PWA metadata — a hint for the OS/browser and accessibility tools.

> **Tip — targeting English users:** change `appConfig.locale.default` to `"en"` AND set `NEXT_PUBLIC_APP_LANG="en-US"` to keep the two in sync.

**⚠️ Email template language:** currently **hardcoded in Indonesian** across all subjects and body copy. Changing `NEXT_PUBLIC_APP_LANG` **does not** translate emails. If your audience is English-speaking, manually edit the template files in `src/shared/email/templates/`.

## 7. System / Auto-managed

### 7.1 `NODE_ENV`

| Property | Value |
|---|---|
| Required | No (Next.js sets it automatically) |
| Exposed | Yes (read-only) |
| Values | `development` / `production` / `test` |
| **DO NOT** set manually | — Next.js handles this |

**Set automatically by:**

- `pnpm dev` → `development`
- `pnpm build` & `pnpm start` → `production`
- Test runners → `test`

**Used in the codebase for:**

- `app/api/auth/callback/route.ts` + `app/api/auth/confirm/route.ts` — detecting localhost for redirect handling (production uses the forwarded-host header)
- `core/auth/components/forgot-password-form.tsx` — dev-only warning logging
- `core/i18n/index.ts` — dev-only missing-translation warning
- `modules/commerce/services/activity.service.ts` — dev-only error logging

**DON'T:** override `NODE_ENV` in `.env.local` or Vercel. Next.js manages it itself. Overriding causes strange bugs (e.g. dev-mode features active in a production build).

## 10. Validation Checklist

Before `pnpm dev` or deploying, verify:

```
[ ] .env.local exists at the project root (not inside src/)
[ ] .env.local is NOT tracked by Git
    → run: git check-ignore .env.local
    → output must be ".env.local" (meaning it's ignored)
[ ] NEXT_PUBLIC_SUPABASE_URL is set, format https://xxx.supabase.co
[ ] NEXT_PUBLIC_SUPABASE_ANON_KEY is set, JWT (~200+ chars)
[ ] SUPABASE_SERVICE_ROLE_KEY is set, different value from the anon key
[ ] ENCRYPTION_KEY is set, base64 32 bytes
[ ] ENCRYPTION_KEY is backed up in a password manager
[ ] RESEND_API_KEY is set, format re_xxxxxxxxxx
[ ] RESEND_FROM_EMAIL is set, format "Name <email@verified-domain.com>"
[ ] SUPABASE_AUTH_HOOK_SECRET is set, format v1,whsec_xxxxxxxxxx (with the prefix!)
[ ] NEXT_PUBLIC_APP_URL is set in production (optional locally)
[ ] Branding vars optional — fill them to match your brand (or keep defaults)
```

### Smoke Test

```bash
# 1. Start the dev server
pnpm dev

# 2. Check the console — no errors should appear like:
#    "NEXT_PUBLIC_SUPABASE_URL is not defined"
#    "ENCRYPTION_KEY env var not set"
#    "SUPABASE_SERVICE_ROLE_KEY not set"
#    "RESEND_API_KEY is not set"

# 3. Open http://localhost:3000
#    - Should redirect to /login
#    - The login page should display your brand name (if NEXT_PUBLIC_APP_NAME is set)
```

A "X not defined" error means the env isn't loaded:

- File naming (`.env.local`, not `.env`)
- Location (project root, next to `package.json`)
- Restart the dev server (Next.js caches env at startup)

## 11. Not Present in the Codebase (Frequently Asked)

- ❌ `DATABASE_URL` — direct Postgres connection. The boilerplate **only uses the Supabase client**; no raw queries via the pg driver. If you need one (for an external migration tool, say), grab it from Dashboard → Project Settings → Database → Connection string. It's not used at runtime.

- ❌ `STRIPE_*` — mentioned in older docs, but the codebase uses **Lemon Squeezy**. LS credentials are stored in the database (encrypted), not in env.

- ❌ `LEMONSQUEEZY_API_KEY`, `LEMONSQUEEZY_WEBHOOK_SECRET` — LS credentials are **per-user**, stored encrypted in `commerce_credentials.encrypted_api_key` and `commerce_webhook_configs.encrypted_secret`. **They're not app-level env vars.** Users enter them in the UI at `/settings/integrations`.

- ❌ `SMTP_*` — SMTP config **isn't used**. All auth emails go through the Send Email Hook + Resend. Custom SMTP in the Supabase dashboard is only a fallback when the hook is disabled.

- ❌ `SENDGRID_*`, `POSTMARK_*`, `MAILGUN_*` — the boilerplate is **Resend-only**. To swap providers, edit `src/shared/email/resend-client.ts` + `send-auth-email.tsx`.

- ❌ `NEXTAUTH_*` — Supabase Auth is used, not NextAuth.
