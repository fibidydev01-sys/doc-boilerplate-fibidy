## Toggle On/Off

Edit `src/config/app.config.ts`:

```ts
modules: {
  admin:    { enabled: true,  path: "/admin" },
  commerce: { enabled: true,  path: "/products" },
  landing:  { enabled: true,  path: "/" },
  saas:     { enabled: false, path: "/workspace" },
  blog:     { enabled: false, path: "/blog" },
  project:  { enabled: false, path: "/projects" },
  forum:    { enabled: false, path: "/forum" },
  chat:     { enabled: false, path: "/chat" },
}
```

### Consequences of Disabling

- Nav links disappear from the sidebar automatically (the boilerplate reads `isModuleEnabled()` in `nav-config.ts`)
- Routes remain accessible if hit directly ‚Äî **delete the relevant route folder** (e.g. `src/app/(marketing)/` for landing, `src/app/(dashboard)/<path>/` for dashboard modules) to block them entirely
- `isModuleEnabled('<name>')` returns `false`

For `landing` specifically: disabling the flag does not delete the public marketing routes (`/`, `/pricing`, `/legal/*`). They live at the route level. To fully strip the marketing surface, delete `src/app/(marketing)/`.

### Consequences of Enabling a Placeholder

If you enable `blog` while the folder is still just a placeholder:

- The sidebar shows the `/blog` link (because nav-config checks `isModuleEnabled`)
- Clicking the link ‚Üí 404 (no route exists yet in `src/app/(dashboard)/blog/`)
- No runtime error ‚Äî but bad UX

**Recommendation:** delete placeholder modules you won't use. Edit:
1. `app.config.ts` ‚Äî remove the entry
2. The `ModuleName` type ‚Äî auto-updates (derived from `modules` keys)
3. Delete the folder `src/modules/<name>/`

## Helper

```ts
import { isModuleEnabled } from "@/config";

if (isModuleEnabled('commerce')) {
  // render commerce-related stuff
}
```

Used in `nav-config.ts` for conditional nav rendering. Use it elsewhere as needed.

## Module Structure (Reference)

Two modules are fully built and serve as references for different concerns:

- **`commerce`** ‚Äî services + API endpoints + DB-backed data flows. Reference for any module that wraps a third-party service (LS, Stripe, Paddle, etc.) with encrypted credentials, webhook ingestion, and dashboard CRUD.
- **`landing`** ‚Äî content data layer + sections + legal templates. Reference for any module that's primarily presentational with env-driven branding interpolation.

## Rule: No Cross-Module Imports

The `commerce` module **may not** import from the `admin` module. The `landing` module **may not** import from `commerce`. If shared logic is needed across 2+ modules:

- **Generic (foundational):** move it to `src/core/`
- **Wrap a 3rd-party service:** move it to `src/shared/`

Enforcement: convention + code review. There's no lint rule that enforces this in the boilerplate (you can add one via an ESLint `no-restricted-imports` rule if you want to be strict).

## Why So Many Placeholders?

Placeholder modules (`saas`, `blog`, `project`, `forum`, `chat`) are intentionally included so that:

1. **The naming convention is clear** ‚Äî if you want to build a SaaS module, the folder name is standardized: `src/modules/saas/`
2. **Config slots are ready** ‚Äî flipping `enabled: true` is all it takes
3. **TypeScript narrowing** ‚Äî the `ModuleName` type already includes every possible module

Trade-off: placeholder folders make the project tree look "unfinished". Delete the ones you don't plan to use if that bothers you.
