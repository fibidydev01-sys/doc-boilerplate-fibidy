## Per-User Architecture

This boilerplate supports **multi-tenant commerce**. Each user can connect their own LS account and sell their own products ŌĆö similar to platforms like Gumroad or Lemon Squeezy itself.

There's no single "admin LS account" ŌĆö every user has their own separate credentials.

### Implications

- `commerce_credentials` ŌĆö one row per user, encrypted API key
- `commerce_webhook_configs` ŌĆö one row per user, per-user webhook URL + secret
- `commerce_orders`, `commerce_subscriptions`, `commerce_customers` ŌĆö scoped by `user_id` via RLS

From Lemon Squeezy's perspective: each user is an **independent merchant** with their own LS account.

### Simplifying to Single-Tenant

If you only need one "central" account (you're the one selling, users only buy):

1. Hard-code an API key in env (replace the per-user credential flow)
2. Skip the `/settings/integrations` UI
3. Use one webhook config via Supabase Hooks / route directly

The boilerplate defaults to multi-tenant ŌĆö stripping that out if you don't need it is on you.

## Security Measures

### Encryption at Rest

Sensitive data is encrypted before insert:

- `commerce_credentials.encrypted_api_key` ŌĆö LS API key
- `commerce_webhook_configs.encrypted_secret` ŌĆö webhook HMAC secret

Uses AES-256-GCM with the `ENCRYPTION_KEY` env var. The `encrypt()` + `decrypt()` helpers live in `src/core/lib/encryption.ts`.

Details in [Database Overview ŌåÆ Encryption](/docs/database/overview#encryption-at-rest).

### HMAC Verification at Webhook

The webhook endpoint verifies the HMAC signature using the user's secret before processing. A webhook without a valid signature ŌåÆ 401 Unauthorized, no DB write.

Implementation uses native Node crypto (`crypto.createHmac("sha256", secret)`) in `src/modules/commerce/lib/webhook-verify.ts`.

### Idempotency

LS may retry a webhook when the response is not 200. The idempotency store (`commerce_webhook_events.event_id UNIQUE`) prevents double-processing:

```sql
INSERT INTO commerce_webhook_events (event_id, user_id, event_type, payload)
VALUES (...)
ON CONFLICT (event_id) DO NOTHING;  -- second time: no-op
```

On conflict, the handler returns 200 without re-upserting orders ŌĆö the webhook is treated as already acknowledged from LS's perspective.

### Service Role Client

The webhook ingestion handler uses `SUPABASE_SERVICE_ROLE_KEY` (bypasses RLS) because webhooks arrive without a user session. The handler is responsible for verifying the token ŌåÆ resolving user_id ŌåÆ writing to that user's rows.

Service role is **only** used in `/api/commerce/webhooks/[token]/route.ts`. Everything else uses the user session + RLS.

## Why Lemon Squeezy, Not Stripe?

**LS is a Merchant of Record (MoR).** LS is the legal entity that sells to the customer, not you. Implications:

| Concern | Stripe (you're the MoR) | Lemon Squeezy (LS is MoR) |
|---|---|---|
| Sales tax / VAT | You collect + remit | LS handles |
| International tax compliance | Painful (EU VAT, US sales tax by state) | LS handles |
| Business registration | Need a legal entity per region | Not needed |
| Chargebacks | Dispute comes straight to you | LS handles, you only lose revenue |
| Fee | 2.9% + $0.30 per txn | 5% + $0.50 per txn |

**Trade-off:** higher fees, but zero compliance overhead. A good fit for solo devs / small teams without an accounting team.

Once the business scales (> $500k/year), switching to Stripe starts making sense on fees alone. At that point, replace the commerce module ŌĆö the pattern is the same (encrypted credentials + webhook); only the API and payload shapes change.
