Email + password is the default provider. Active out of the box. **All flows ship** — signup, login, forgot password, reset password.

## Status

✅ **Fully implemented.**

| Flow | Page | Form Component |
|---|---|---|
| Login | `/login` | `LoginForm` → `EmailPasswordForm` |
| Signup | `/register` (when `allowPublicSignup: true`) | `RegisterForm` → `EmailPasswordRegisterForm` |
| Forgot Password | `/forgot-password` | `ForgotPasswordForm` |
| Reset Password | `/reset-password` | `ResetPasswordForm` |

## Supabase Setup

Active by default in new projects. Verify:

1. Supabase Dashboard → **Authentication** → **Providers** → **Email**
2. Make sure **Enable Email provider** is ✓

### Important Settings

| Setting | Dev | Production |
|---|---|---|
| Confirm email | Off (login immediately) | **On** (prevent fake accounts) |
| Secure email change | Off | **On** |
| Secure password change | Off | **On** |

### `requireEmailVerification` Must Match

If you set **Confirm email: On** in Supabase, update the config too:

```ts
// src/config/app.config.ts
auth: {
  requireEmailVerification: true,  // ← match what's in Supabase
}
```

The setting in app config is only a hint for UI logic (display "check your email", disable auto-login). Actual enforcement happens in Supabase.

## Login Flow

```
1. User submits email + password at /login
2. signInWithPassword({ email, password })
3. Supabase verifies credentials → sets the session cookie
4. The boilerplate verifies the profile in user_profiles (exists + active)
5. If verification fails → signOut + error to UI
6. If it succeeds → log activity (user.login) + role-aware redirect
```

Source: `src/core/auth/components/providers/email-password-form.tsx`.

## Signup Flow

Depends on `requireEmailVerification`:

### Case 1: `requireEmailVerification: false` (Auto-login)

```
1. User submits the /register form
2. signUp({ email, password, data: { full_name } })
3. DB trigger handle_new_user() auto-creates a row in user_profiles
4. Session is set immediately
5. Log activity (user.signup, auto-login=true)
6. Redirect to the dashboard
```

### Case 2: `requireEmailVerification: true` (Confirm first)

```
1. User submits the form
2. signUp(...) — Supabase sends a confirmation email via the hook
3. UI: "Check your email to verify"
4. User opens the email → clicks the link
5. The link goes to /api/auth/confirm?token_hash=xxx&type=signup&next=/dashboard
6. The confirm route verifies the OTP → establishes the session → verifies the profile → redirects
```

The `user.signup` activity log in Case 2 happens in the callback route (after the session exists), not during the signUp call (because RLS blocks inserts without a session).

Source: `src/core/auth/components/providers/email-password-register-form.tsx`.

## Forgot Password Flow (PKCE)

```
1. User enters email at /forgot-password
2. resetPasswordForEmail(email, { redirectTo: /api/auth/callback?next=/reset-password })
3. Supabase sends the email with a link
4. User clicks the link → lands at /api/auth/callback?code=xxx&next=/reset-password
5. The callback exchanges the code → establishes the recovery session → redirects to /reset-password
6. User submits a new password → updateUser({ password })
7. Log activity (user.password_reset_completed) → redirect to the dashboard
```

### Anti-Enumeration

The `/forgot-password` form **always** shows a success message (even if the email doesn't exist). This prevents enumeration attacks.

### Why via the Callback Route?

`redirectTo` MUST go to `/api/auth/callback` (not directly to `/reset-password`). With `@supabase/ssr` PKCE flow, the email link carries a `code` param that must first be exchanged via `exchangeCodeForSession`. If it went straight to the reset page, the recovery session wouldn't be established → the form would get stuck with "invalid session".

Source:

- `src/core/auth/components/forgot-password-form.tsx`
- `src/app/api/auth/callback/route.ts`
- `src/core/auth/components/reset-password-form.tsx`

## Reset Password Flow

```
1. Land on /reset-password (from the callback redirect)
2. The component checks that a session exists (getSession)
3. No session → show "Reset link is invalid or expired"
4. Session exists → show the new-password form
5. Submit → updateUser({ password })
6. Log activity → redirect to the dashboard
```

## Validation Schema

Client-side validation uses Zod + i18n:

```ts
// src/core/lib/validators.ts
export const loginSchema = z.object({
  email: z.string()
    .min(1, t("auth.validation.emailRequired"))
    .email(t("auth.validation.emailInvalid")),
  password: z.string()
    .min(1, t("auth.validation.passwordRequired"))
    .min(6, t("auth.validation.passwordMinLength")),
});

export const registerSchema = z.object({
  fullName: z.string().min(1).min(2).max(100),
  email: emailSchema,
  password: z.string().min(1).min(8),
  confirmPassword: z.string().min(1),
}).refine((data) => data.password === data.confirmPassword, {
  message: t("auth.validation.passwordMismatch"),
  path: ["confirmPassword"],
});
```

Error messages are auto-translated via `react-hook-form` + `zodResolver`.

**Password policy:**

- Login: minimum 6 characters (lenient; existing users may have old passwords)
- Signup: minimum 8 characters (stricter for new users)

Customize in `src/core/lib/validators.ts`.

## Disable Public Signup

For internal-only usage (no public signup):

```ts
// src/config/app.config.ts
auth: {
  allowPublicSignup: false,
}
```

Effects:

- The `/register` page auto-redirects to `/login`
- The "Sign up" link in LoginForm is hidden
- Admins provision users manually (via the Supabase Dashboard or a custom admin panel)

## Disable the Email Provider

```ts
// src/config/app.config.ts
passwordProviders: ["magic-link"] as const,  // email removed
```

The LoginForm automatically hides the **Password** tab. Users can only sign in via OAuth or Magic Link.

## Troubleshoot

### "Invalid login credentials"

- Wrong email/password → ask the user to reset their password
- The user signed up but hasn't confirmed email → if `requireEmailVerification: true`, they must confirm first

### Password reset link "invalid or expired"

- The link has already been used (single-use)
- Link expired (default 1 hour)
- `redirectTo` in `resetPasswordForEmail` is wrong (must be `/api/auth/callback`, not `/reset-password` directly)
- Supabase Dashboard Redirect URLs don't include the callback route

### Signup succeeds but the email doesn't arrive

- Send Email Hook isn't set up → falls back to Supabase default SMTP (3/hour limit, often spammed)
- Resend domain isn't verified
- See [Email Troubleshooting](/docs/troubleshooting#emails-still-not-arriving)
